# 10 — File Storage

## 1. What needs storing

| Asset | Source in UI | Notes |
|---|---|---|
| Business **logo** | app splash/login `logo-full.png`; owner avatar in headers | one per business |
| Business **hero photo** | microsite hero ("Hero photo — salon interior" placeholder) | one per business |
| **Gallery images** | microsite `GALLERY` grid (4 cells) | many per business, ordered |
| **Staff/barber avatars** | initials today; real photos likely | one per staff |
| **QR code PNG** | `QRSheet.tsx` "Download"/"Share" for `tejotime.com/{slug}` | generated, cacheable |
| **"Map" / location image** | microsite map placeholder | optional |
| (Future) invoices/receipts | subscription billing | PDF |

The current apps use **local bundled assets and CSS gradient placeholders** — there is no upload yet. The backend introduces managed storage for all of the above.

## 2. Storage backend

- **S3-compatible object storage** (AWS S3, Cloudflare R2, or Supabase Storage).
- Buckets (or prefixes): `logos/`, `heroes/`, `gallery/`, `avatars/`, `qr/`, `receipts/`.
- Served via **CDN** (CloudFront / Cloudflare) with long cache TTLs and cache-busting via content-hash filenames.
- Path convention: `{assetType}/{businessId}/{uuid}.{ext}` — tenant-scoped keys.

## 3. Upload flow (presigned, direct-to-storage)

```
1. Client → POST /api/v1/uploads/sign
     { assetType: "gallery", contentType: "image/webp", byteSize: 240000 }
2. Server validates: allowed assetType, MIME allow-list, size ≤ limit, owner role
   → returns { uploadUrl (presigned PUT), fileKey, publicUrl, expiresIn }
3. Client → PUT the bytes directly to uploadUrl (bypasses the API)
4. Client → PATCH the owning resource with { logoUrl | heroImageUrl | ... } = publicUrl
   (server verifies the key exists, was uploaded by this tenant, and is within the tenant prefix)
```
Never accept arbitrary external URLs for profile/gallery fields — only keys minted by `/uploads/sign` for that `business_id` (prevents SSRF/hotlinking and enforces tenancy).

## 4. Validation & processing

| Rule | Value |
|---|---|
| Allowed image MIME | `image/jpeg`, `image/png`, `image/webp` |
| Max image size | 5 MB (configurable) |
| QR PNG | server-generated (see §6), not user-uploaded |
| Post-upload processing (async job) | validate real MIME (magic bytes), strip EXIF/GPS, generate resized variants (thumb/card/hero), optionally re-encode to WebP |
| Virus/abuse scan | optional ClamAV / provider scan on upload |

Image variants (e.g., `gallery` → `thumb 320w`, `card 640w`, `full 1280w`) generated by a `media.process` background job and referenced by the client for responsive loading.

## 5. Access control

- **Public assets** (logo, hero, gallery, QR, avatars) — world-readable via CDN (they render on the public microsite). Keys are unguessable UUIDs.
- **Private assets** (receipts/invoices) — served via short-lived signed URLs, owner-only.
- Deletion: removing a gallery image deletes the object (or tombstones it) and its variants.

## 6. QR code generation

- On business creation / slug change, generate a QR PNG (and SVG) encoding `https://tejotime.com/{slug}`, store at `qr/{businessId}.png`, expose via `GET /business/qr` ([05 §B](./05-api-endpoints.md#b-business--settings--business)).
- Regenerate on slug change; cache aggressively (immutable per slug).
- "Download" and "Share" in `QRSheet.tsx` use the stored PNG URL + deep link.

## 7. Retention & privacy

- Deleting a business or exercising erasure (NFR-PR3) removes all objects under its tenant prefixes.
- Strip location EXIF from uploaded photos by default (privacy).
- Lifecycle rules: expire orphaned uploads (signed but never attached) after 24h.
